# 00 AI总规则手册

> 本文件是整个项目约束 AI 行为的最高优先级手册。任何模块开发、重构、补丁、调试、优化都必须先服从本文件。

## 0.1 项目一句话定义

本系统是一个：

**多租户私域渠道商城平台（Channel Commerce Platform）**

不是淘宝式流量平台，不是 ERP 附属页面，也不是单商家商城。

## 0.2 不可变项目事实

AI 必须永远记住以下事实：

1. 平台支持多个租户（商家）
2. 一个租户可有一个或多个店铺
3. 客户只能访问被分享的小程序店铺
4. 默认没有“逛平台所有商家”的入口
5. 商品可能来自 ERP、第三方系统、平台自建
6. 商品必须支持 SKU、多属性、多规格、不同价格、不同库存
7. 不做支付，**下单即生成订单**
8. 订单生成后由平台异步推送到 ERP / 第三方系统
9. 系统必须支持“无外部系统商家”
10. 系统必须支持未来接入多个不同 ERP 或其他系统

## 0.3 AI 工作总原则

### 原则1：先建模，后编码
顺序必须是：

```text
业务边界 -> 数据模型 -> API契约 -> Service设计 -> 编码 -> 测试 -> 验收
```

### 原则2：任何代码都不得脱离多租户上下文
除平台级管理表外，绝大多数业务实体必须携带：

- tenant_id
- shop_id
- created_at
- updated_at

### 原则3：任何接口都不得跳过服务层
架构必须是：

```text
Router/API
  -> Service
    -> Repository / ORM
      -> Database
```

### 原则4：不可把来源商品直接当渠道商品
必须维持双层模型：

```text
source_product -> channel_product
```

### 原则5：订单推送必须异步任务化
不能在“下单成功的主事务”里直接硬推外部系统。

## 0.4 输出要求

AI 每次输出必须包含下列信息之一：

- 数据模型
- 迁移脚本
- 接口定义
- 服务层逻辑
- 单元测试
- 集成测试
- 验收说明

禁止只给“概念性描述”却不给落地结果。

## 0.5 回复风格要求

AI 输出必须：

- 结构化
- 明确文件路径
- 明确模块边界
- 明确依赖关系
- 明确本次修改影响范围
- 明确后续待办

禁止：

- 一次性改太多模块
- 未经说明修改核心架构
- 把多个业务领域混在一个文件
- 用“先这样凑合”式代码

## 0.6 每次任务的固定开场动作

AI 接到任务后必须先做这 4 件事：

1. 判断任务属于哪个领域模块
2. 检查是否触发多租户 / 店铺 / SKU / 推送等核心约束
3. 列出将要新增或修改的文件
4. 说明任务是否会影响现有数据结构或 API 契约

## 0.7 每次任务的固定收尾动作

AI 完成任务后必须附带：

1. 本次新增/修改文件清单
2. 本次新增/修改模型清单
3. 本次新增/修改接口清单
4. 测试建议
5. 下一步推荐任务

## 0.8 优先级冲突规则

当以下内容冲突时，按此优先级处理：

1. 本手册
2. 系统蓝图与边界
3. 数据库与领域模型规范
4. API与服务层规范
5. 当前任务说明
6. 临时口头补充要求

如果 AI 发现冲突，必须先指出，再给兼容方案。
